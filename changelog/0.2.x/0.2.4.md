# 0.2.4 — 2026-06-15

## Added

- **`DEVOPS_STATUS_DISABLE_ACTIVE_PROBES`** — when `true`, the arbitrary-target probe tools `devops_check_dns` and `devops_check_certs` are omitted from the registered tool surface; the five vendor-registry/incident tools (`devops_list_vendors`, `devops_status_check`, `devops_get_incidents`, `devops_watch_stack`, `devops_suggest_action`) remain. Defaults to `false` (both registered — no behavior change). For shared/public multi-tenant instances. ([#3](https://github.com/cyanheads/devops-status-mcp-server/issues/3))
- **`ACTIVE_PROBE_TOOL_NAMES`** — exported `ReadonlySet<string>` from `src/mcp-server/tools/definitions/index.ts`, the single source of truth for the gated probe set consumed by the `src/index.ts` surface filter.

## Changed

- **Server instructions** drop the "`devops_check_certs` and `devops_check_dns` work for any domain" line when `DEVOPS_STATUS_DISABLE_ACTIVE_PROBES` is set, so the orientation matches the registered surface.
- **`release:github`** script invokes `bun run` instead of `tsx`.

### Dependencies

- `@biomejs/biome` ^2.4.16 → ^2.5.0
- `vitest` ^4.1.8 → ^4.1.9

## Fixed

- **`allowPrivateTargets` dead field** — `src/utils/ssrf-guard.ts` now reads `getServerConfig().allowPrivateTargets` instead of `process.env.DEVOPS_STATUS_ALLOW_PRIVATE_TARGETS` directly; the config-layer field is no longer bypassed. ([#4](https://github.com/cyanheads/devops-status-mcp-server/issues/4))

## Security

- **`DEVOPS_STATUS_ALLOW_PRIVATE_TARGETS`** schema tightened from `z.string().optional()` to `z.enum(['true', 'false']).default('false').transform((v) => v === 'true')` — an invalid value (e.g. `yes`) now throws a `ConfigurationError` at startup naming the env var instead of silently resolving to `false`. ([#4](https://github.com/cyanheads/devops-status-mcp-server/issues/4))
- **Probe-tool gating** lets operators withhold the two caller-controlled outbound-probe tools (DNS queries and TLS handshakes to arbitrary hosts/ports) on shared instances, removing a reconnaissance vector and IP-reputation liability from the server. ([#3](https://github.com/cyanheads/devops-status-mcp-server/issues/3))
